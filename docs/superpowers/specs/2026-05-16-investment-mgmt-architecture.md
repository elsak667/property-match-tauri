# 招商管理模块数据架构方案

> **目标：** 在现有飞书多维表格基础上，构建统一的客户管理体系，支撑招商全流程（线索→客户→跟进→承租）

## 一、当前现状

| 已有表 | 用途 | 状态 |
|--------|------|------|
| 招商线索表 | 入口，全员提交线索 | 已建 |
| 客户信息表 | 客户主数据 | 已建 |

## 二、目标架构（2 + 1 表）

```
飞书 Bitable App: 招商管理
  ├ 招商线索表      — 入口（全员可提交）
  ├ 客户信息表      — 主表（含承租信息）
  └ 跟进记录表      — 日志（新建）
```

**设计原则：**
- 线索表和客户表独立（数据来源不同）
- 跟进记录单独建表（一对多，方便按时间/人员筛选）
- 联系人嵌入客户表（变更少，不需要独立列表）
- 承租信息嵌入客户表（核心字段，需要快速查看）

## 四、工作流

```
[线索提交] → 管理员审核 → 分配招商员
                              ↓
                        [线索转化] → 新建客户
                                       ↓
                              [招商跟进] → 跟进记录
                              [承租管理] → 更新承租信息
                                       ↓
                              [阶段推进]
```

**关键流程：**
1. **线索提交**：任何人可通过表单提交，系统记录
2. **线索转化**：状态变为"已转化"时，在客户表新建记录
3. **招商跟进**：在客户详情面板新增跟进记录
4. **承租更新**：客户承租信息变更时直接编辑客户表

## 五、前端页面规划

```
招商管理 Tab
  └ 客户列表页
       ├ 顶部：搜索框 + 阶段筛选 + 新建客户按钮
       ├ 列表：名称 | 行业 | 招商人员 | 阶段 | 承租载体 | 操作
       └ 详情面板（右侧滑出）
            ├ Tab1: 基本信息（含承租信息、联系人）
            ├ Tab2: 跟进记录（列表 + 新增）
            └ Tab3: 进度历史（只读日志）

线索管理 Tab
  └ 线索列表页（管理后台）
       ├ 状态筛选
       ├ 列表：企业名 | 联系人 | 招商人员 | 状态 | 时间
       └ 操作：审核 → 分配 → 转化

线索提交入口（全员可用）
  └ 首页/导航 → 线索表单（独立页面）
```

### 技术细节

- 空字符串写入Bitable date/number 字段会导致500错误，用 `...(value && { field: value })` 模式修复
- CORS: `mode: "cors"` 必须在所有fetch调用中设置
- CustomerForm pre-fill: 通过 `initialData` prop 传递已有数据，优先于 `getCustomer()` 加载
- 阶段变更后需手动更新列表中的 `selectedCustomer` state
