# 招商管理模块实施计划

**项目：** property-match-tauri（浦发集团招商平台）
**创建日期：** 2026-05-14
**状态：** 待启动

---

## 一、整体架构

### 1.1 技术选型

| 层级 | 技术方案 | 说明 |
|------|---------|------|
| 前端 | React + TypeScript + Zustand | 复用现有技术栈 |
| 后端 | Cloudflare Worker | 复用现有基础设施，Token 安全性高于纯前端直连 |
| 数据存储 | 飞书多维表格（Bitable） | 由 Worker 代为读写 |

### 1.2 数据流

```
前端 → Worker API → 飞书 Bitable API → 飞书多维表格
         ↓
      本地缓存（KV）
```

### 1.3 Worker API 设计

**线索相关：**
| 接口 | 方法 | 说明 |
|------|------|------|
| `/invest-api/clues` | GET | 获取线索列表 |
| `/invest-api/clues/:id` | GET | 获取线索详情 |
| `/invest-api/clues` | POST | 提交新线索（公开） |
| `/invest-api/clues/:id` | PUT | 更新线索状态 |
| `/invest-api/clues/:id` | DELETE | 删除线索 |
| `/invest-api/clues/:id/convert` | POST | 线索转为客户 |

**客户相关：**
| 接口 | 方法 | 说明 |
|------|------|------|
| `/invest-api/customers` | GET | 获取客户列表 |
| `/invest-api/customers/:id` | GET | 获取客户详情 |
| `/invest-api/customers` | POST | 新建客户 |
| `/invest-api/customers/:id` | PUT | 更新客户 |
| `/invest-api/customers/:id` | DELETE | 删除客户 |
| `/invest-api/customers/:id/stages` | PUT | 更新招商进度 |
| `/invest-api/customers/:id/visits` | GET/POST | 拜访记录 |
| `/invest-api/customers/:id/progress` | GET | 招商进度历史 |

## 二、阶段划分

### 第零阶段：招商线索管理（入口）
**目标：** 作为招商体系入口，全员可提交线索，审核后跟进，招商落地后转为正式客户

**交付物：**
- [ ] 飞书多维表格设计（线索表）
- [ ] 前端：线索提交表单（公开入口）
- [ ] 前端：线索管理后台（审核、跟进）
- [ ] 线索状态流转（待核实 → 核实中 → 意向确认 → 已签约/已失效）
- [ ] 奖励条件判定（是否可享受一个月租金奖励）
- [ ] 线索转客户逻辑

**线索表字段设计：**

| 字段 | 类型 | 说明 |
|------|------|------|
| clue_id | 自动编号 | 主键 |
| company_name | 文本 | 企业名称 |
| contact_name | 文本 | 联系人姓名 |
| contact_phone | 文本 | 联系电话 |
| source_recommender | 文本 | 推荐人 |
| source_recommender_phone | 文本 | 推荐人联系方式 |
| required_area | 数字 | 需求面积（㎡） |
| preferred_district | 文本 | 意向区域 |
| target_property | 关联 | 目标物业载体 |
| investment_staff | 文本 | 负责招商员（人工分配） |
| status | 单选 | 线索状态 |
| reward_eligible | 布尔 | 是否有奖励资格（由载体判断） |
| reward_note | 文本 | 奖励备注 |
| created_at | 日期 | 提交时间 |
| updated_at | 日期 | 更新时间 |

**线索状态（简化版）：**
1. 待核实
2. 跟进中
3. 已转化（招商落地，转为客户）
4. 已失效

**奖励条件：**
- 一个月租金作为奖励
- 非所有项目都有奖励，需在备注中说明原因
- 转为客户后仍需记录奖励状态

**线索转客户逻辑：**
- 只有状态为「已签约」的线索才能转为正式客户
- 转化时自动填充客户信息表（从线索表带入基础字段）
- 转化后线索表保留记录，标记状态

## 三、多维表格结构总览

| 表格名称 | 说明 | 关联 |
|---------|------|------|
| 招商线索表 | 入口表，全员可提交，审核后跟进 | 转化后 → 客户信息表 |
| 客户信息表 | 主表，存储客户基本信息和招商状态 | 联系人表、拜访记录表、进度变更日志 |
| 联系人表 | 客户的关键联系人 | customer_id → 客户信息表 |
| 拜访记录表 | 每次拜访的详细记录 | customer_id → 客户信息表 |
| 进度变更日志 | 记录进度变更历史（自动） | customer_id → 客户信息表 |

## 五、技术债务与扩展预留

1. **飞书 API 限流应对**：Worker 层加缓存，避免频繁调用
2. **数据迁移预留**：当前不涉及，但联系人、拜访记录支持后续补充字段
3. **批量操作**：未来可扩展批量录入、批量更新功能
4. **消息推送**：未来可扩展飞书消息通知（如新客户录入自动通知招商员）

```
┌─────────────────────────────────────────────────────────────┐
│                        招商线索提交                          │
│  （全员可提交，填写企业信息 + 联系方式 + 需求）                 │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│                     招商管理员审核                            │
│           （核实信息 → 分配招商员 → 进入跟进流程）              │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│                    线索状态流转                              │
│  待核实 → 跟进中 → [已转化 / 已失效]                         │
│                                    ↓                         │
│                              奖励条件判定                     │
│                         （由目标载体判断是否符合奖励）         │
└─────────────────────┬───────────────────────────────────────┘
                      ↓（已签约）
┌─────────────────────────────────────────────────────────────┐
│                      转为正式客户                             │
│         （自动带入线索中的企业信息 + 奖励状态）                 │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│                   客户招商进度追踪                            │
│  初步接触 → 需求确认 → 实地看房 → 谈判中 → 签约入驻            │
└─────────────────────────────────────────────────────────────┘
```
